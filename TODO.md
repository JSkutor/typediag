### 구체적인 저장 방법

일단 page 스키마에서 created_at 를 좀 세분화 하자. 시작하는 시점이랑 끝나는 시점 두 개로 나눠.
run 을 끝내는 조건을 생각해보자.
실시간으로 끝나는걸 확인할 수는 없어. 왜냐면 입력하다가 그냥 끄면 끝인데 이걸 감지를 못함. 아마도?
시간 경과로 알아야됨.
page 에 처음글자 입력을 시작할 때:
가장 최근 run 객체를 확인:
run이 in progress 이고, page 시점이 최근(3분이내)이면:
그 run을 fk로 하는 page 생성하고 진행.
pending 상태라면:
그 pending run을 삭제하고, 지금 시점으로 pending run을 생성하고 진행.
run 이 in progress 이고, 마지막 page 의 끝나는 시간이 지금으로부터 3분보다 이전이면:
(끝나는 시간이 있는 page는 마지막 page가 아닐 수도 있음.)
그 page를 마지막으로 하고, 이후에 끝나는 시간이 없는 page는 삭제.
그리고 run 의 status를 completed로 바꾸고 run을 마무리.
run의 끝나는 시점도 마지막 page의 끝나는 시점으로 함.
status가 completed 이거나 run이 없으면 pending run을 새로 만들고 시작.

넘어갈 수 있는 조건부터 생각을 해야됨.
스페이스나 엔터로 넘어가려면 끝까지 입력을 해야됨.
정확도 수치가 있으니까 글자수만 맞추면 스페이스, 엔터로 넘어가지도록.
그리고 스킵이 있어야함. 그냥 그 글을 입력하기 싫을 수도 있음.
이거는 오른쪽 방향키로. 방향키로 왔다갔다 하는 로직은 이미 있지 않나.
좀 더 구체화 해보자.
이미 지난 page를 왼쪽 방향키로 돌아간다면 같은 글을 다른 page 객체로 다시 새로 시작하는것처럼.
오른쪽 방향키는 그냥 스킵. 입력을 하는 순간 page가 생기는거니까 아무 작업 안해도 되지 않나.

page가 완료될 때(스킵 아님):
시작 시점이 오래 전이면(5분보다 이전):
run을 새로 만들어서 fk 교체.
이전 run은 completed로 마무리.
run이랑 page의 시작 시간은 현재에서 긴 공백(latency) 이후의 latency를 다 합친 값을 뺀 값으로 설정.
가장 최근 run 객체를 확인:
pending 이라면:
run을 in progress 로 바꾸고, fk를 그 run으로 지정 후 진행.
in progree 라면:
정상적으로 진행.

접속했을 때, 로컬 스토리지를 정리하자. 이전에 안끝난 page는 없애고, run도 마무리.
이러면 page 입력이 글의 중간부터 시작되는 경우는 사라짐.
3분보다 이전인 경우만 run 마무리. 최근이면 page는 지우되, run은 유지.



### tokey statistics
- 모래시계
  4가지 경우로 분류: good -> good, bad -> bad, g->b, b->g
  g가 위, b가 아래 라고 하면 시간의 흐름에 따라 직선을 그으면 모래시계 모양임.
  개선되고 있는 정도를 나타냄.
  여기에는 좀 복잡한 로직 넣어도 될듯.
  선형회귀 넣으면 좋을거 같음.
  간단함. 기울기 구하는거임.

  일단 선형회귀는 그냥 raw 데이터 이용해서 하면 됨. 기울기 구하고.
  데이터 개수가 적을 때는 막아야됨.
  finalUpperBound (50개 미만일 때 저장 x)
  
  분절회귀. 이러면 모래시계 모양은 아니긴 한데 상관없음. 이게 더 좋을듯.
  함수랑 같이 실제 데이들도 도트로 보여주는 게 좋을거 같음. 다는 아니고 좀만.
  
  방정식을 반환하는 독립적인 모듈 만들어야됨.
  그리드 서치를 통해 c 초깃값을 설정하고,
  무제오 알고리즘(Muggeo's Method)을 통해 최종적으로 c값을 도출.
  그리고 최소제곱법을 통해 방정식 도출 후 반환.

    입력:
    interface KeyEvent {
    fromKey: string | null;
    toKey: string;
    latencyMs: number;
    keyChar?: string;
    holdDurationMs?: number | null;
    isCorrect?: boolean | null;
    expectedChar?: string | null;
    }
    이 배열에서 toKey 랑 isCorrect 만족하는 데이터 필터.

    import pwlf
    import numpy as np
    x = np.arange(len(data))
    y = np.array([2,4,6,8,10,11,12,13])
    my_pwlf = pwlf.PiecewiseLinFit(x, y)
    c0 = my_pwlf.fit(2)

    시간순 그대로 X,Y 각각 2차원 행렬로 변환.
    X = [[x1, max(0, x1 - c0), I(x1 > c0)], [], ...]
    Y = [[y1], [y2], ...]
    (duration은 여기선 안쓰지만 활용도 좋을거 같음.)

    c0, X, Y 를 입력으로 무제오 알고리즘 -> c 도출
    
    c로 최소제곱법(OLS) -> 방정식 도출
    
    

  그리고 좀 대강 분석해서 텍스트로 설명도 넣어주고.
  


- 오타 유발율:
  tokey랑 isCorrect false 값으로 필터링 해서, 그 이전 쌍이 correct 인지 탐지.
- db 탐색 잘 되면 늦게 눌러서 키 순서가 바뀌어서 오타를 낸 경우 카운트
  incorrect 두 쌍이 있고, 뒤에 쌍이 이거임.
  앞의 쌍의 target이 이 tokey임.

  - 가장 많이 치는 순서쌍 top5: tokey 쪽에 통계로. (#3)

  - 시프트를 생략하고 원래 연속적인 정답 상의 키 순서와 비교해서 시프트 때문에 얼마나 느려졌는지
