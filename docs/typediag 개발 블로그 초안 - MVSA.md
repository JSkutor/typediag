# MVSA
---

TL;DR (요약)
- **MVSA(Maximum Valid Sequence Aligner)**는 한글의 2차원적 조합 특성을 고려해 사용자의 의도를 최대한 반영하는 실시간 자소 단위 오타 판별 알고리즘이다.
- **핵심 설계**: Stateless 구조를 유지하면서, 내부적으로는 이중 역방향 탐색(R2L)과 패닉 모드를 통해 사용자의 타건 의도(REPLACE, OMIT 등)를 휴리스틱하게 추론한다.
- **최적화**: 역방향 탐색의 맹점(과거 상태 유실)을 극복하기 위해 Prefix 시뮬레이션을 도입했으며, 앵커(Anchor) 기반 캐싱과 점진적 연산을 통해 $O(N^2)$ 연산을 상수 시간에 가깝게 최적화했다.
- **결과 취합**: 자소 단위의 판별 결과는 투표(Voting) 메커니즘과 연산자 우선순위를 거쳐 최종적으로 사용자에게 보여지는 글자 단위의 상태로 렌더링된다.

## 1. 도입

>MVSA는 입력의 자소 단위 오타 여부를 실시간으로 판단하는 함수다.

일반적인 오타 판별 로직은 **사용자의 의도**를 완전히 반영하지 못한다.
어절 중간에 추가 오타가 하나 입력되면, 이후 글자들까지 모두 오타로 분류되는 상황이 흔히 일어난다.
`가나다라`를 `간다라` 라고 입력한 상황을 가정해본다.
`ㅏ` 를 하나 누락했을 뿐인데 정타로 인식되는 글자가 아예 없을 것이다.
이런 문제를 해결하면서 한글 조합 과정의 자연스러움을 더하기 위해 직접 알고리즘을 고안했다.
최대한 많은 입력을 정타로 분류하기 위한 목적을 녹여 **MVSA(Maximum Valid Sequence Aligner)**로 이름 지었다.
암베사라는 애칭으로 부르면 발음하기 쉽다.

정타를 최대한 살린다는 개념을 생각해보면 떠오르는 알고리즘이 하나 있을지도 모른다.
CS에서 DP(Dynamic Programming)를 다룰 때 필수적으로 소개되는
**LCS(Longest Common Subsequence)**, **레벤슈타인 거리(Levenshtein Distance)**와 관점이 상당히 비슷하다.
레벤슈타인 거리는 문자열을 다른 문자열로 바꾸는 데 필요한 최소의 편집 연산(삽입, 삭제, 대체) 횟수를 구한다.
STREET -> TREATS 의 경우 'S'의 삭제, 'E'의 대체, 'S'의 삽입으로 총 3을 출력한다.
레벤슈타인 거리를 어절 단위 스코프, 한글 도메인 특화 함수로 만든 것이 MVSA 라고 할 수 있다.
실제로 MVSA의 출력은 `EQUAL`, `PARTIAL`, `REPLACE` 등의 편집 연산 라벨(이하 op, Operator)이다.

## 2. 문제 정의

### 조합 중 상태
한글은 자모를 조합하는 언어다.
한 개의 글자가 여러 개의 자소로 구성되는 2차원적 특징을 갖는다.
그럼에도 **오타 판별은 글자 조합 중에도 실시간으로 일어나야 한다.**
완성된 글자와 비교해서도 안되고, 조합 중인 상태를 무조건 맞거나 틀리게 판단해도 안된다.
`한` 을 입력하려고 할 때,
`ㅎ`,
`하`,
`한`
이 세 가지 상태는 모두 정타로 처리돼야 한다.
그리고  `학` 이 되는 순간 비로소 오타로 판별해야 한다.


### 도깨비불 현상
조합 문제와 이어지는 내용이다.
한글에서는 두 글자를 입력하는 과정 중에,
두 번째 글자의 초성이 이전 글자의 종성(받침)으로 붙는 상태를 필연적으로 갖는다.
직관적인 이름은 아닌 것 같지만, 이런 현상을 도깨비불 현상이라고 부른다고 한다.
오타의 관점에서 해결해야 할 문제는 다음과 같다.
**정타가 다음 글자 초성으로 인해 일시적으로 받침이 생기면서 오염되어도, 이를 오타로 판별해서는 안된다.**
그렇다고 이전에 친 글자는 막아버려서 도깨비불 현상 자체를 막는 것도 옳지 않다.
한글 IME(입력기)에서 도깨비불 현상은 자연스럽게 허용돼야 하기 때문이다.

또 그렇다고 해서 정타로 한 번 완성된 글자를 영원히 정타 상태로 두는 것도 잘못됐다.
`가나다` 를 치는 상황에서,
`간` 을 치면 정타지만,
`각` 을 치면 오타다.
한글적인 관점에서 보면 한 글자의 오타 판별을 하려면 다음 글자도 봐야 한다는 뜻이다.
이는 밑에서 **탐색 윈도우**와 **R2L(Right-to-Left, 역방향) 탐색**으로 해결한다.


### 오타 전파
다음은 위치 문제다.
어절에서의 오타 전파 예시는 앞에서 말했던 `간다라` 상황이다. (`가나다라` 가 정타.)
**오타가 글자수에 영향을 미치면 뒤의 글자들도 따라서 오타로 밀려 처리되기 쉽다.**
정답 문장과 글자 단위로 일대일 비교한다면 문장 끝까지 영향이 갈 수 있기 때문에 특히 위험하다.

## 3. 개발기

MVSA의 개발 순서는 다음과 같았다.
- 오랜 삽질 끝에 MVSA를 독립적으로 구축
- 다듬기
- 대규모 구조 리팩터링
- 다듬기

이렇게 크게 두 번의 주요 개발 페이즈가 있었고, 둘 다 정상적이지는 않았다.

초반 개발에서는 기획이 없었던 것이 문제였다.
처음 MVSA를 적용한 커밋은 1,000라인 규모의 대형 커밋이 되고 말았다.
여기서 대부분의 삽질이 있었지만 커밋을 안 쪼갠 탓에 잘 기록되지는 않았다.
고칠 게 끊임없이 생겨서 프로젝트에서 토큰을 특히 많이 썼던 부분이다.

후반부 개발은 시기가 문제였다.
블로그를 쓰기 시작한 시점까지는 MVSA가 그냥 안정적인 상태라고 생각했다.
지금 보면 뭔 자신감으로 테스트를 그렇게 적게 했는지 모르겠다.
결국 블로그 퇴고 중 로직의 근간을 갈아엎는 리팩터링을 해야 했다.
앞으로 말하는 리팩터링은 이 후반부 대규모 리팩터링을 뜻한다.

리팩터링의 방향은 결과적으론 맞는 방향이었다.
이전에는 연산이 자소 단위와 글자 단위를 넘나들며 일어났다.
또한 **근본적으로 비교는 글자 단위인 반면, 입력은 자소 단위였던 것이 맞지 않았다.**
이 문제로 계속 엣지 케이스들을 만나고 처리하는 과정이 반복되었다.

그러다 직관을 믿고, **연산의 기본 단위를 자소 단위로 고정**하는 대대적인 수정을 감행했다.
이 결정 자체도 쉽지 않았다.
에이전트가 특히 성능을 못 내던 부분이었기 때문이다.
버그들을 잡는 과정에서 최대한 근본적으로 해결하기 위해 다방면으로 질문을 많이 했었다.
그리고 답변에서는 찬양과 비판이 주사위 굴리듯 나왔다.
모델별로도 말이 많이 달라서 직감으로 밀고 나가는 수밖에 없었다.

특히 이런 버그들이 결정타를 쳤다.
- `태양` + 백스페이스 한 번 -> `탱` (`태야` 가 맞다.)
- `인류` -> `읺류` 입력 시 오타 판별
계속 글자 단위 비교로 처리가 힘든 부분에서 문제가 나오고 있었다.
이런 이유로 관심사 분리와 함께 리팩터링을 진행했다.
여차하면 git restore 할 마음가짐이었다.

먼저 stateless, 캐싱과 같은 MVSA의 거시적인 특성을 설명한다.

## 4. 기초 설계

### 관심사 분리

초반 개발에서는 LCS 기반 유틸리티를 독립적인 함수로 대체하는 작업이 있었다.
알고리즘과 오타를 표시하는 UI 렌더링 프론트 부분도 완전히 분리했다.

문제는 이 MVSA 코드가 900줄의 mvsa.ts 단일 파일이었다는 것이다.
리팩터링에서 자소 단위 비교로 전환함과 동시에 두 파일로 쪼갰다.

#### mvsaCore.ts
입력을 받아 모든 단일 키 입력(자소 단위)에 대해 오타를 판별해 분류 라벨(Operator)을 부여한다.

Core에서의 모든 연산은 qwerty 물리 키, 즉 자소 단위로 수행된다.
**이 파일에서는 글자라는 개념이 존재하지 않는다.**
하지만 분명 한글 특성상 글자로의 조립은 필수적이다.
한글은 1개의 글자가 2~3개의 자소를 갖는 2차원식 형태이기 때문이다.
자소 단위의 오타 판별 결과를 글자 단위로 변환하지 않고 사용자에게 보여주면 안된다.

여담으로, `es-hangul` 라이브러리를 이번에 처음 알게 되었다.
`es-hangul` 이 한글 입력을 영타로 치환하는 기능을 완벽하게 제공한다.
Toss가 사내에서 쓰다가 오픈소스로 공개한 라이브러리라고 한다.

이 파일에 MVSA의 거의 모든 핵심 로직이 들어있다.
다음 `5. 오타 판별` 섹션에서 설명한다.

#### mvsaAggregator.ts
Core의 출력을 받아, 보이는 글자 단위로 병합한다.

Core와 반대로 Aggregator는 자소 단위 판별 결과를 글자 단위로 취합시킨다.
먼저 자소 입력들이 각각 **어떤 target 글자를 의도한건지** 알아낸다.
그리고 최종적으로 개별 글자가 정타인지 오타인지 판단한다.
여기서는 투표와 연산자 우선순위 시스템을 이용한다.
`6. 자소 병합` 섹션에서 설명한다.



### Stateless 구조
**Stateless 함수**는 이전 입력(상태)을 기억하지도 않고,
외부 상태를 직접 변경하지도 않는 순수한 독립적인 함수를 의미한다.

MVSA를 stateless 함수로 설계하면 얻는 이점은 다음과 같다.
**1. 백스페이스를 로직으로 구현할 이유가 없어진다.**
	타자를 치는 input과 지우는 input에 대해 MVSA 입출력이 같다.
	매 입력마다 새로 계산을 하기 때문에, 백스페이스로 글자를 지운다면 그 글자는 있었던 적이 없게 된다.
**2. 어절 격리 속성을 유지하면서 어절 간 넘나드는 입력 또는 백스페이스를 자연스럽게 처리한다.**
	다른 서비스들과 마찬가지로, Typediag도 스페이스를 기준으로 입력을 분리하는 속성을 갖는다.
	만약 상태 기반이었다면, 이전 어절 판단을 취소하고 다시 판단하는 추가적인 로직이 필요했을 것이다.

트레이드 오프도 있다.
이전 상태를 저장하지 않기 때문에, 매 입력마다 이전 입력들이 다시 중복 연산되는 문제다.
이 부분은 캐싱의 도입으로 해결한다.


### 캐싱 (Caching)
Stateless 속성과 다르게 캐싱 기능은 MVSA 도입 후 나중에서야 적용되었다.
메인 기능도 버그가 계속 나오는 상황에서 최적화는 뒷전이었기 때문이다.
에이전트한테 따로 언급을 안했더니 구조가 괴상해지고 있었다.

MVSA는 input으로 이때까지 입력된 키를 모두 받는다.
그리고 그 각각의 키들에게 오타 판별 결과값을 op(`EQUAL`, `REPLACE` 등)로 부여한다.
하지만 지금 수정이 일어나고 있는 어절이 아니라면,
이미 과거에 끝낸 앞부분 입력까지 다시 연산할 이유는 전혀 없다.
그럼에도 이전의 로직은 키를 입력할 때마다 이전의 결과값을 모두 버리고 처음부터 다시 연산했다.
너무 stateless한 상태였어서 생긴 문제라고 할 수 있다.
결과적으로 매 입력마다 이전 부분까지 다 계산하므로 $O(1 + 2 + ... + n) = O(n^2)$ 의 비효율성을 갖는다.

이를 해결하기 위해 캐시를 도입했다.
입력은 여전히 처음부터 입력한 키 배열을 모두 받는다.
하지만 이제 이전에 연산을 했던 부분은 **캐시 히트 처리돼서 다시 연산하지 않는다**.
캐시도 어절 단위로 저장된다.
이 시점에서 한 번의 입력에서 연산 비용은 $O(w + k)$ 이다.
w은 이전까지의 어절 수, k는 현재 입력 중인 어절의 키 입력 수를 의미한다.

애초에 한 페이지에 글이 많지는 않아 원래 로직에서도 프레임 드랍까지는 보이지 않았다.
그래서 더 미리 알지 못했다.

```typescript
// 어절 단위의 자소 상태를 고유한 Cache Key로 생성
const isCompleted = endQIdx < inputChars.length;
const cacheKey = `${tTokIdx}:${qIdx}:${wordInputChars.join("")}:${isCompleted}`;

let wordResults: JasoAlignResult[];

if (this.cache && this.cache.has(cacheKey)) {
  // 이미 연산이 끝난 이전 어절들은 O(1)로 캐시 히트
  wordResults = this.cache.get(cacheKey)!;
} else {
  // 현재 입력 중이거나 변경이 발생한 어절만 재연산
  wordResults = this.alignWordIncrementalHeuristic(wordTargetTokens, wordInputChars, /* ... */);
  this.cache?.set(cacheKey, wordResults);
}
```

## 5. 오타 판별 (mvsaCore.ts)

MVSA 함수의 stateless 속성과 최적화 방법에 대해 알아봤다.
다음은 MVSA의 핵심이 되는 mvsaCore.ts의 로직에 대해 설명한다.
크게 패닉 모드와 시뮬레이션 도입을 다룬다.
가장 중요한 부분이다.

### Normal Mode
본질적인 로직을 이해하기 위해 캐싱 기능은 잠시 없다고 생각한다.

MVSA는 stateless 하기 때문에 매 입력마다 전체 target에 대한 input을 모두 다시 받는다.
ex) `abc de` 의 `e` 입력에서 이전의 `abc de` 입력을 모두 받고 오타 판정을 한다.
`a` 부터 순서대로 순회하며 input과 target을 비교한다.
비교는 모두 알파벳으로 치환된 자소 단위로 진행된다.

**정타는 qwerty 단위(자소 단위)로 일대일 비교를 통해 쉽게 처리된다**.
하지만 MVSA는 오타를 판별하고 처리하는 방식이 일반적이지 않다.
불일치를 확인 시 다른 특별한 액션이 필요하다.
이러한 필요에서 Normal/Panic Mode를 설계했다.

모드 간 전환의 목적은 사용자의 의도를 최대한 파악해 정타로 반영하는 것이다.
오타를 그냥 오타로 보지 않는다.
뭘 빠뜨린건지, 뭘 잘못 쳤는지, 뭘 추가로 눌렀는지를 판단한다.

평소 오타가 없는 normal 상황에서는 자소 단위로 정답과 비교해나간다.
그러다 오타를 감지하는 순간 패닉 모드로 전환한다.
전환 후에는, 매 입력마다 Normal Mode로의 복구 가능 지점을 탐색한다.
복구 지점의 의미는 말 그대로 사용자가 다시 정타를 치기 시작한 위치다.
오타의 시작 부분부터 복구 지점 사이에 매치가 안된 입력들은 `REPLACE`, `OMIT` 등의 오타로 분류된다.

### Panic Mode

#### 탐색 윈도우
input과 target을 비교하며 복구 지점을 찾는 것이 mvsaCore.ts의 첫번째 목적이다.
복구 지점 탐색도 역시 자소 단위로 이루어진다.

복구 지점 탐색 로직이 LCS 와의 가장 큰 차이를 만든다.
매 입력마다 LCS를 돌려 복구 지점을 찾고자 했으면 자연스럽지 않아진다.
같은 글자가 있는 어절이나 특이한 글자 패턴의 엣지 케이스를 전혀 방어하지 못한다.

이 부분을 해결하기 위해 target text 의 탐색 범위를 직관적인(휴리스틱) 방법으로 제한했다.
**엔진이 순회하고 있는 자소보다 한 칸 더 내다본다**.
유저가 오타 상태에서 입력한 자소 수보다 하나 더 target text에서 탐색한다는 뜻이다.
탐색 범위 제한은 다음 전제를 바탕으로 한다.
건너뛰는 오타(`OMIT`)가 있어도 실수로 두 키 이상을 뛰어넘지 않는다.
이 강한 휴리스틱이 R2L 역방향 탐색을 가능하게 한다.

사실 '순회 중인 자소보다 한 칸 더'라는 말이 이 시점에선 구현되기 힘들다.
이 점은 Prefix Simulation 파트에서 해결한다.



#### 이중 역방향 루프
구체적인 탐색 방법을 설명한다.

패닉 이후부터의 비교는 input과 target 다대다 비교이므로, 이중 루프를 기반으로 한다.
여기서 **바깥쪽 루프와 안쪽 루프의 탐색 방향을 모두 R2L 역방향**으로 구현했다.
위의 탐색 윈도우 제한 조건 때문에 가능했던 로직이고, 둘이 시너지를 갖는다.
자세한 이유를 설명하겠다.


MVSA는 stateless 함수다.
백스페이스는 따로 처리할 필요가 없다.
따라서 모든 로직은 오타를 고치지 않고 그냥 넘어가는 걸 전제해야 한다.
이런 관점에서 보면 과거 입력과 현재 입력 중 어느 곳을 더 신뢰해야 하는지는 명확하다.
과거는 이미 틀린 타건일 확률이 높고, 현재 입력이 다시 정타일 확률이 높다.

그렇기 때문에 역방향 탐색을 통해 현재 입력과 타겟을 먼저 확인해야 한다.
탐색 윈도우 조건 덕분에 현재(커서)와 과하게 먼 미래의 타겟과 비교하지 않는다.

또한 정방향으로 하게 되면 우연히 얻어걸려 판단이 잘못되는 상황이 있을 수 있다.
연속으로 오타를 치고 있을 때, 최신 오타와 과거 타겟이 일치해 매칭된다면 줄줄이 잘못 밀려버린다.

```typescript
// 바깥 루프 (Input): 입력된 버퍼를 뒤에서부터(R2L) 거꾸로 읽어온다.
// 가장 최근에 타이핑한 마지막 자소(미래)부터 먼저 검사하기 위함이다.
for (let pIdx = wordInputChars.length - 1; pIdx >= qIdx && !matchFound; pIdx--) {
  const pChar = wordInputChars[pIdx];

  // 안쪽 루프 (Target): 타겟의 탐색 범위 역시 뒤에서부터(R2L) 읽는다.
  // 반복되는 글자 구간에서 앞쪽(과거) 글자에 우연히 오매칭되는 케이스를 피하기 위함이다.
  for (let lookTIdx = targetLookaheadEnd - 1; lookTIdx >= tIdx; lookTIdx--) {
    
    // 가장 먼저 발견되는 일치 지점(복구 지점)을 찾습니다.
    if (wordTargetTokens[lookTIdx].char === pChar) {
      matchFound = true;
      bestMatchInputIdx = pIdx;     // Input의 앵커(Anchor)
      bestMatchTargetIdx = lookTIdx; // Target의 앵커(Anchor)
      break; 
    }
    
  }
}
```

mvsaCore.ts에는 글자 단위 개념이 없기 때문에 알파벳으로 예시를 들겠다.
타겟: `a b d d`
입력: `a b c d`
먼저 사용자의 의도를 어떻게 판단해야 할지 정해야 한다.
3번째 입력이자 오타인 `c` 에 대해 가능한 판단은 다음과 같다.
1. 실수로 추가로 입력한 오타
2. `d` 를 누르려다 잘못 `c` 를 누른 오타

둘 다 말은 된다.
하지만 MVSA는 `REPLACE` 오타라고 판단한다.
`REPLACE` 오타가 `INSERT` 오타보다 훨씬 흔하게 일어나기 때문이다.
인접한 다른 키를 누르거나 인지적 착각으로 다른 키를 누르는 경우가 대부분이다.
이 부분은 정답은 없기 때문에 **mvsa의 로직 철학으로 정의**하고 밀고 나가야 했다.

```text
                 [Target 순회]
                    < - - -
 Target:   a    b    d    d
                          ^
                          | (매치)
                          |
 Input :   a    b    c    d
                     ^
                     |
                 (REPLACE)
```

물론 타겟의 마지막이 `d` 가 아니었다면 `d` 와 `d` 끼리 잘 매치되면서 `c` 는 `INSERT` 로 판단된다.

또 `OMIT` 오타를 감지하기도 역방향이 더 적합하다.
역방향 순회 중 복구 지점을 찾고, 아직 순회를 안 한 앞부분을 모두 적절히 오타 처리하는 것이 로직의 의도다.
```text
                 [maxLookahead 탐색 범위]
                   < - - - - - - -
 Target:   a       b       c       d       e
           ^       ^       ^      ^ 
           |       |       |     /
           |  (REPLACE) (OMIT) /
           |       |         / 
 Input :   a       a       d 
```


만약 타겟을 순회하는 루프(안쪽 루프)가 정방향이었으면 판단은 바뀐다.
`d` 입력이 타겟의 왼쪽 `d` 랑 매치된다.
그러면서 `c` 는 mvsa 철학과 맞지 않게 `INSERT` 로 처리된다.
```text
                 [Target 순회]
                    - - - >
 Target:   a    b    d    d
                      ^
                       \ (매치)
                        \
 Input :   a    b    c    d
                     ^
                     |
                 (INSERT)
```


실제로 이렇게 예시를 들어가면서 에이전트한테 설명하고 이해시키려 애먹었던 기억이 난다.
심지어 core와 aggregator가 분리되기 이전엔 한글 자소를 예시로 설명해야 했다.
글자 단위 연산이 섞여 있어서 알파벳으로 비유를 할 수가 없었기 때문이다.
추론 좋은 에이전트도 가끔 대놓고 틀릴 정도로 LLM은 한글 자소에 취약하다.
`인류` -> `읺류` 같은 케이스에서 뜬금없이 `ㅂ` 을 꺼내오곤 했다.



### Prefix Simulation
패닉 모드에서 mvsaCore.ts의 기능적인 부분은 모두 다뤘다.
오타를 판별하는 방법에 대해서는 대강 그림이 그려질 것이다.
하지만 입력이 차근차근 쌓여가는 모습을 상상하고 있었다면, 한 가지 놓치고 있는 부분이 있다.

#### 문제 인식: 과거 상태 기억
MVSA는 stateless 속성 때문에 한(마지막) 입력 시, 이전 입력들을 모두 순회하며 재계산 해야 한다.
여기서 R2L 역방향 탐색이 문제가 된다.
이중 역방향 탐색에서는 입력과 타겟의 마지막과 마지막을 먼저 비교한다.
따라서 이전 자소들을 순회하고 있을 때는 순회 지점이 마지막이라고 인식되야 한다.
그래야 **순회를 함으로써 입력이 차근차근 쌓이는 효과**를 낼 수 있다.

하지만 어쩌다가 개발 중간에 코드가 엇나갔다는 걸 발견했다.
모든 순회 지점에서 마지막 입력을 현재 자신이 아닌, 실제 입력의 끝으로 보게 바뀌어 있었다.

```text
[버그 상황: B 순회에서 B가 아닌 미래의 입력을 타겟과 비교.]

Target:   a    b    c    d
                 ^
                  \ (미래의 입력을 비교)
                   v
Input :   a    B    C
               ^
           (현재 순회)
```


사실 이전에 해결했어야 하는 문제가 늦게 터진 것이다.
구상은 되어 있으니까 구현만 하면 될 것 같지만 상황은 생각보다 어려웠다.
**이전 순회에서 기억해야 할 것들이 많아지기 때문**이다.
넘겨받아서 유지해야 할 상태값(state)들이 복잡하다.
패닉이 시작된 지점, 패닉 버퍼(패닉 입력 임시 배열), 자소 싱크, 그리고 모드 상태 관리까지 필요할 수도 있다.
매 순회에서 빡센 상태 관리 로직을 요구한다.
결과적으로는 이게 맞는 방향일 수도 있었지만, 굳이 이렇게 할 필요 없다고 생각했다.
본질은, 순회 시 이전 순회 지점의 상태를 아는 것이다.


#### 해결: 입력 시뮬레이션
근본적으로, 패닉 모드를 구상할 때, stateless를 간과하고 있었던 탓이라 생각했다.
버퍼에 입력이 차근차근 쌓인다고 상상하고 있었기 때문이다.
실제로는 이런 실시간적인 흐름이 아니라, 과거를 다시 계산하는 것이 포인트다.
그래서 이 처음 생각으로 돌아가는 방법을 생각해냈다.
**실제로 입력이 단계적으로 쌓이면서 진행되는 상황을 인위적으로 만들었다**.

**기억하기보단 반복**하겠다는 과감한 결정이었다.
기존의 1-pass $O(n)$ 순회 대신, 매 입력에서 $O(n^2)$ 이 나오는 구조로 바꾸는 결정이다.
입력 규모가 한정되어있고, 중복 계산은 나중에 최적화 하면 된다는 생각이었다.

```text
[입력 'abcd' 중 마지막 'd'를 타건했을 때의 엔진 순회 비교]

기존: 단일 순회 (1-Pass)
한 번의 루프로만 훑고 지나가기 때문에, Step 간 상태 정보 전달 필수.

   Step 1:  a
   Step 2:  ·    b
   Step 3:  ·    ·    c
   Step 4:  ·    ·    ·    d


해결: 입력 시뮬레이션 (Prefix Simulation)
과거부터 현재까지 타건이 쌓여가는 과정을 루프 안에서 완벽히 재현함.

   Step 1:  a
   Step 2:  a ── b
   Step 3:  a ── b ── c
   Step 4:  a ── b ── c ── d
```

이렇게 매 입력마다 앞부분(prefix) 시뮬레이션을 돌려 실제로 입력이 쌓여가는 구조를 구현했다.
한 번의 순회마다 과거 상황을 모두 재현하는 것이다.
이러면 복잡한 상태 로직이 필요 없이, 그대로 지금 순회 안에서 필요한 정보를 도출할 수 있다.
순회(Step) 간 정보를 보내는 게 어려울 뿐, 같은 루프(Step) 안에서는 쉽기 때문이다.
연산비용을 내주고 stateful 속성을 얻은 것이다.

반면, 단점은 명확하다.
캐싱 없이는 한 입력 마다, 자소 수 $n$ 에 대해 $O(n^2)$ 의 연산이 발생하는 위험한 구조다.
전체 문장을 타건하는 과정을 누적하면, 중복 연산이 무려 $n^3$ 번에 가깝게 일어난다.

문장이 길어봤자 100자 정도다 하는 생각으로 밀었다.
중복 연산 부분은 나중에 충분히 보완할 수 있다고 생각했다.

이 구조는 앵커를 도입하면서 진가를 발휘한다.

#### 최적화: 점진적 연산
중복 연산 문제는 **점진적 연산(Incremental Computation)**을 이용해서 해결했다.
이미 계산된 부분은 그대로 두고 새로 추가되거나 변경된 부분만 추가로 연산하는 기법이다.
시뮬레이션 상황과 직관적으로 맞는 방법이다.

구현을 위해 먼저 **앵커(Anchor)**를 도입했다.
매 시뮬레이션에서 마지막 EQUAL 판정을 받은 자소가 앵커가 된다.
전달받은 결과의 마지막 자소가 EQUAL이 아니라면 현재 패닉 모드에 있다는 뜻이다.
패닉 모드에서서는 패닉 버퍼에 있는 모든 자소에 대해 연산이 필요할 수도 있다.
따라서 시뮬레이션에서는 앵커 이후만을 연산한다.


시뮬레이션 흐름
1. 직전 결과값을 가져온다.
2. 가져온 결과에서 앵커를 찾는다.
	앵커 이전은 확정된 상태라고 보고 재계산하지 않는다.
3. 직전 결과에서 앵커 이후만 떼어낸 것과 새로 들어온 입력 하나를 합친다.
4. 이 구간을 재계산한다.
5. 결과를 앵커 이전 부분과 합쳐서 다음 시뮬레이션으로 넘긴다.

```text
[ 직전 결과 (Step n-1) ]
 ┌──────────────────────┬──────────────────┐
 │ 앵커 이전 (EQUAL 확정)  │ 앵커 이후 (패닉)     │
 └──────────────────────┴──────────────────┘
			            ▲ 앵커 (마지막 EQUAL 지점)
			            │
			            ├─ 1. 구간 분리 (Split)
			            │
 ┌──────────────────────┐   ┌──────────────┐   ┌───────────┐
 │ [ 재사용 구간 ]         │   │ [ 패닉 구간 ]  │ + │ [신규 입력] │ 
 │ (재계산 생략)           │   │ 직전 오타 상태  │   │ n번째 자소  │
 └──────────────────────┘   └──────────────┘   └───────────┘
			│	                             │
			│	                             ▼
			│	                     1. 부분 재연산 (MVSA)
			│	                     (패닉 구간 + 신규 입력)
			│	                             │
            ├─ 3. 결과 병합 (Merge) ───────────┘
            ▼
 ┌─────────────────────────────────────────┐
 │ [ 새로운 결과 (Step n) ]                   │
 │ 다음 시뮬레이션으로 전달                      │
 └─────────────────────────────────────────┘
```

이 전체 시뮬레이션은 매 키 입력에서 진행된다.
다만 이전 어절은 캐싱을 통해 제외되어, 시뮬레이션은 현재 입력 중인 어절 안에서만 가볍게 동작한다.



예시:
타겟: `aaa bb cdddf`
입력: `aaa bb cdeef`
마지막 `f` 를 입력한 상황이다.
`a` 와 `b` 구간은 어절 단위로 캐시 히트 처리된다.

```text
Target:  c ── d ── d ── d ── f
Input:   c ── d ── e ── e ── f
         │    │              │
Step 1:  ⚓   ·    ·    ·    ·   시뮬: c
Step 2:  ·    ⚓   ·    ·    ·   시뮬: cd
Step 3:  ·    ⚓   ·    ·    ·   시뮬: cde (e 패닉)
	          ↑        
	         [앵커 고착]
	         
Step 4:  ·    ⚓   ·    ·    ·   시뮬: cdee (패닉 유지)
Step 5:  ·    ·    ·    ·    ⚓  시뮬: cdeef (f 복구)

Input:   c ── d ── e ── e ── f
              ↑              ↑
           [기존⚓]        [신규⚓]

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
캐시:    [██████] ║ [░░░░░░░░░░░]
	 이전 확정 구간 ║ 재계산 구간
       (연산 생략) ║ (기존 패닉 + 신규 입력)
```


결과적으로 $O(N^2)$ 구조를 극복하고, 선형 시간으로 최적화됐다.
이전 어절들의 개수($w$)와 재연산되는 자소 개수($k$)에 대해 $O(w + k)$ 까지 최적화했다.
이제 연산 비용은 실질적 상수 시간이 된다.
둘 다 거의 한 자리수에서 노는 값들이므로 성능 문제는 완전히 해소된다.


```typescript
// mvsaCore.ts - Prefix Simulation (점진적 연산) 핵심 로직 발췌
for (let i = 1; i <= wordInputChars.length; i++) {
  const prefix = wordInputChars.slice(0, i); // 1글자, 2글자... 입력을 점진적으로 늘림
  
  // 1. 이전 시뮬레이션 결과에서 마지막으로 확정된 정타 지점(Anchor) 탐색
  const lastEqualIdx = results.findLastIndex(r => r.op === "EQUAL");
  
  // 2. 앵커 이전 구간은 '확정'으로 간주하고 재계산 생략
  const confirmedResults = lastEqualIdx === -1 ? [] : results.slice(0, lastEqualIdx + 1);
  
  // 3. 앵커 이후의 '패닉 구간'과 '신규 입력'만 합쳐서 부분 재연산
  const tailResults = this.alignWordHeuristic(remainingTarget, remainingQwerty);
  
  // 4. 결과 병합 후 다음 단계(Step n+1) 시뮬레이션으로 전달
  results = [...confirmedResults, ...tailResults];
}
```

## 6. 자소 병합 (mvsaAggregator.ts)

앞에서 패닉 모드, 시뮬레이션으로 MVSA가 어떻게 동작하는지 거의 다 확인했다.
이제 힘을 조금 빼고 프론트에 필요한 로직을 다룬다.

### 목적
목적은 단순하다.
**mvsaCore의 결과값을 글자 단위로 병합**하는 것이다.
자소 단위 오타 판단값을 글자 단위의 판단값으로 승격시킨다.

MVSA가 자소 단위 비교로 리팩터링 되면서 관심사가 분리되었다.
위에서 다룬 mvsaCore는 최종 출력값으로 모든 자소에 판단값 라벨(Operator)를 붙여 전달한다.
다음과 같은 값을 가질 수 있다.
`EQUAL`, `REPLACE`, `INSERT`, `OMIT`, `PENDING`

이 출력값을 프론트에서 그대로 활용할 수는 없다.
`가` 를 `나` 라고 쳤다고 자음만 빨갛게 칠하지는 않기 때문이다.

이 함수는 두 가지의 기능을 한다.
1. 입력과 타겟을 글자 단위로 매칭시킨다.
2. 최종적으로 그 글자의 op를 판단한다.

이 두 기능은 각각 투표 매커니즘, op 우선순위를 통해 작동한다.

### 투표 매커니즘 (Voting)
MVSA는 제일 처음 작업으로 타겟을 자소 단위로 쭉 분리한다.
이 때 각각의 자소에 `vCharIndex` 를 달아주어, 원래 어떤 글자 출신인지를 기억한다.
이 정보를 mvsaCore에서 알아낸 입력-타겟 매칭 정보와 합친다.
```json
[
	// 0번째 글자 "안"에서 나온 자소들 (vCharIndex: 0)
	{ "char": "ㅇ", "jasoIndex": 0, "vCharIndex": 0 },
	{ "char": "ㅏ", "jasoIndex": 1, "vCharIndex": 0 },
	{ "char": "ㄴ", "jasoIndex": 2, "vCharIndex": 0 }
]
// -->
[
	// "안" (vCharIndex 0) 매칭 결과
	{ "op": "EQUAL", "char": "d", "targetChar": "ㅇ", "targetJasoIndex": 0, "targetVCharIndex": 0, "inputIndex": 0 },
	{ "op": "EQUAL", "char": "k", "targetChar": "ㅏ", "targetJasoIndex": 1, "targetVCharIndex": 0, "inputIndex": 1 },
	{ "op": "EQUAL", "char": "s", "targetChar": "ㄴ", "targetJasoIndex": 2, "targetVCharIndex": 0, "inputIndex": 2 }
]
```

참고로 한글의 자소, 글자간 전환은 앞서 언급한 `es-hangul` 라이브러리를 적극 활용한다.

투표 매커니즘은 예시로 명확히 보여진다.
타겟: `가나`
입력: `간`
타겟의 `가` 와 `나` 를 각각 0번, 1번 글자라고 인덱싱 한다.
```json
[
  // 0번째 타겟 글자 "가" (vCharIndex: 0)
  { "char": "ㄱ", "jasoIndex": 0, "vCharIndex": 0 },
  { "char": "ㅏ", "jasoIndex": 1, "vCharIndex": 0 },

  // 1번째 타겟 글자 "나" (vCharIndex: 1)
  { "char": "ㄴ", "jasoIndex": 2, "vCharIndex": 1 },
  { "char": "ㅏ", "jasoIndex": 3, "vCharIndex": 1 }
]
```

입력의 `ㄴ` 은 mvsaCore에서 타겟의 `나` 의 글자의 일부로 판단되어 EQUAL을 받는다.
위에서 말한 입력-타겟 정보이다.
```json
[
  // 입력: "간"의 'ㄱ(r)'과 'ㅏ(k)'
  { "op": "EQUAL", "char": "r", "targetChar": "ㄱ", "targetJasoIndex": 0, "targetVCharIndex": 0, "inputIndex": 0 },
  { "op": "EQUAL", "char": "k", "targetChar": "ㅏ", "targetJasoIndex": 1, "targetVCharIndex": 0, "inputIndex": 1 },
  
  // 입력: "간"의 'ㄴ(s)' -> 타겟 '나'의 초성 'ㄴ'과 매칭
  { "op": "EQUAL", "char": "s", "targetChar": "ㄴ", "targetJasoIndex": 2, "targetVCharIndex": 1, "inputIndex": 2 },
  
  // 타겟 '나'의 남은 자소 'ㅏ'는 아직 입력되지 않음
  { "op": "PENDING", "char": "", "targetChar": "ㅏ", "targetJasoIndex": 3, "targetVCharIndex": 1 }
]
```

`ㄴ` 은 다음의 정보를 갖게 된다.
출신 타겟 글자 (`나`)
op (`EQUAL`)

```text
+-------------------------------------------------------------------+
| [ 타겟 글자 ]             가 (1번 타겟)                 나 (2번 타겟)    |
+-------------------------------------------------------------------+
                         /             \                    |
                       /                 \                  |
+-------------------------------------------------------------------+
| [ 자소 & OP ]       ㄱ                  ㅏ                ㄴ         |
|                  (EQUAL)             (EQUAL)           (EQUAL)    |
+-------------------------------------------------------------------+
                       \                 /                  |
                         \             /                    |
                           +---------+----------------------+
                                            |
+-------------------------------------------------------------------+
| [ 입력 글자 ]                               간                       |
|                                                                   |
| [ 투표 결과 ]          (1번 타겟: 2표 / 2번 타겟: 1표) => '가' 매칭       |
+-------------------------------------------------------------------+
```

입력을 기준으로 어떤 타겟에 매칭되는지 계산한다.
`간` 에서 `ㄱ` 과 `ㅏ` 는 0번 글자 출신이다.
반면 `ㄴ` 은 1번 글자 출신이다.
**입력 글자를 구성하는 자소들이 자신의 출신을 말한다(표를 던진다)**.
2:1의 결과로 입력 `간` 은 타겟 `가` 와 매칭된다.


동점이라면 커서와 가깝고 미래의 글자에게 우선순위를 둔다.
```text
      [과거(이미 친 글자)]       📍(현재 커서)   [미래(쳐야 할 글자)]

순위:      ... 6등 < 5등 < 4등  📍  1등 > 2등 > 3등 ...
위치:   ... (-3)   (-2)   (-1) 📍 (+1)  (+2)  (+3) ...
```

동점이 나오는 예시는 다음과 같다.
타겟: `박사`
입력: `바가`
`가` 입력에서 `ㄱ` 과 `ㅏ` 가 서로 다른 글자 출신이므로 1:1 동점이다.
'바'는'박'에 매칭되면서 '가'를 입력하는 시점에서 입력 `바` 는 과거 입력이 된다.
따라서 -1의 위치를 갖게 되고, 안좋은 우선순위를 부여받는다.
반면, `사` 는 아직 입력되지 않았으므로(되는 중이므로) `ㅏ` 는 +1의 위치를 갖는다.
이렇게 `ㅏ` 의 출신인 `사` 와 `가` 가 매칭된다.




### 연산자 판별 (Operator)
타겟과 입력의 자리는 정해졌다.
이제 입력 글자에 색을 입힐 차례다.
자소 단위인 오타 판별 **결과를 글자 단위로 올려 오타를 빨갛게 표시**해야 한다.

op를 글자 단위로 바꾸는 방법은 단순하다.
**글자를 구성하는 자소들의 op중 우선순위가 가장 높은걸 채택**한다.

`REPLACE` > `INSERT` > `EQUAL` > `OMIT` > `PENDING`

`REPLACE` 가 가장 높은 우선순위를 갖는다. 
이미 초성 등을 다른 걸로 잘못 입력했을 때의 경우 때문이다.
거기서 없던 받침을 추가로 치든, 있어야 할 받침을 빠뜨리든 이건 `REPLACE` 가 되어야 한다.

우선순위가 약할수록 조건이 빡빡하다는 뜻이 된다.
`OMIT` 은 `EQUAL` 인 자소도 포함되면 안되므로 `EQUAL` 보다 낮다.
`PENDING` 도 마찬가지로 모든 자소가 `PENDING` 이어야 한다.
사실 이 둘은 경합하지 않는다.
커서 기준으로 `OMIT` 은 과거, `PENDING` 은 미래를 나타내는 op이기 때문이다.

이 순위를 먼저 적용하고 후처리를 한번 하면 끝이다.
입력된 글자와 타겟 글자가 정확히 일치하지 않으면 `EQUAL` 에서 `PARTIAL` 이라는 op로 바꾼다.
이 후처리가 없으면, 맞게 쓰다 말고 스페이스로 넘어갔을 때 오타 처리가 안된다.
`EQUAL` 만 한두 개 남아있을 것이기 때문이다.

## 7. 마무리
자소 병합 섹션에서 트러블슈팅 과정이 없는 것은, 의외로 쉽게 끝났기 때문이다.
왜냐하면, 투표 매커니즘과 op 우선순위 아이디어가 모두 에이전트 것이었다.
별거 아니지만 경각심이 드는 부분이다.
직접 제안한 로직에서는 삽질을 몇 시간 했던걸 생각하면 에이전트를 다시 보게 된다.

MVSA는 프로젝트를 구상하고 있던 당시에는 깊게 생각하지 못했던 부분이었다.

로직까지는 미리 생각하지 못했지만 요구사항은 가지고 있었는지도 모른다.
어느정도 타협이 있었다면 만들어질 수 없었던 함수다.
